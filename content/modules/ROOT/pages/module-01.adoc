== Objective

Detect *{target_cve}* on the application server, contain the vulnerable `httpd` service, and harden the system posture -- all driven by Event-Driven Ansible responding to Splunk alerts.

== Scenario

A Splunk saved search detects that `httpd` on `rhel01` is vulnerable to {target_cve} (Apache httpd SSRF, CVSS 9.8). No patch is available yet. Your immediate priority is to limit exposure while maintaining service availability.

== Exercises

=== Exercise 1.1: Scan the System

Run the *Scan System* job template in AAP Controller. This playbook checks `yum updateinfo` for known CVEs affecting installed packages.

Review the output -- you should see {target_cve} flagged against the installed `httpd` version.

=== Exercise 1.2: Contain the Service

Run the *Contain Service* job template. This restricts `httpd` to internal networks using firewalld zones and hardens SELinux booleans.

Verify that `httpd` is still running but only accessible from internal networks.

=== Exercise 1.3: Harden the Posture

Run the *Harden Posture* job template. This deploys security headers, enables AIDE file integrity monitoring, and tightens system configuration.

=== Exercise 1.4: Run the Workflow

Run the *Vulnerability Containment* workflow template, which chains all three steps: Scan -> Contain -> Harden.

=== Exercise 1.5: Trigger via EDA

Simulate a Splunk webhook alert to trigger the containment workflow automatically through Event-Driven Ansible.

== Summary

You have detected a critical CVE, contained the vulnerable service without taking it offline, and hardened the system posture. In the next module, you will apply the patch when it becomes available.
