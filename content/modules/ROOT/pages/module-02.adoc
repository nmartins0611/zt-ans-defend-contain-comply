= Module 2: CONTAIN -- Policy-Gated Patching

== Objective

Apply advisory *{target_advisory}* through a policy-gated workflow with pre-patch validation, an approval gate, post-patch verification, and compliance reporting.

== Scenario

Red Hat has published {target_advisory} to address {target_cve}. Before applying the patch, your organization requires that all policy gates pass and a security team member approves the change.

== Exercises

=== Exercise 2.1: Pre-Patch Policy Check

Run the *Pre-Patch Check* job template. This validates five policy gates:

1. Maintenance window
2. Backup verification
3. Disk space
4. Service health
5. Change approval status

Review the output to see which gates pass.

=== Exercise 2.2: Apply the Patch

Run the *Patch System* job template. This applies {target_advisory} using `dnf` from the errata repository on `central`.

=== Exercise 2.3: Post-Patch Verification

Run the *Post-Patch Verify* job template. This confirms that {target_cve} is no longer detected and the patched `httpd` version is installed.

=== Exercise 2.4: Generate Compliance Report

Run the *Report Compliance* job template. This generates an HTML compliance report capturing the full remediation lifecycle.

=== Exercise 2.5: Run the Policy-Gated Workflow

Run the *Policy-Gated Patching* workflow template. This chains: Pre-Patch Check -> Approval Node -> Patch System -> Post-Patch Verify -> Report Compliance.

Approve the workflow when prompted.

== Summary

You have applied a security patch through a fully auditable, policy-gated workflow with approval controls. In the next module, you will containerize the hardened application.
