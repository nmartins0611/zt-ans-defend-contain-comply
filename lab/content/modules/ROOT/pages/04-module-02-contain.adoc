= Module 2: CONTAIN — Patch and Enforce
:source-highlighter: rouge
:toc: macro
:toclevels: 1

toc::[]

Red Hat has published **RHSA-2024:5138** — a security advisory addressing CVE-2024-38476 in Apache HTTP Server. A patched `httpd` package is now available in your repositories.

However, Meridian Financial's change management policy requires that patches pass several gates before application: backup verification, maintenance window check, approval, and post-patch compliance validation.

In this module you will build and execute a policy-gated patching workflow in AAP Controller.

== Learning objectives

By the end of this module, you'll be able to:

* Write an Ansible playbook that queries OPA for policy decisions
* Construct a multi-node AAP workflow with approval gates
* Execute a patching workflow with policy enforcement
* Generate compliance evidence as a workflow artifact

== Exercise 1: Examine the Patch Policy

[NOTE.why]
====
*Why we do this:* In regulated environments, you can't just apply patches whenever you want. Change management policies exist because rushed patches cause outages. OPA (Open Policy Agent) codifies these policies so they can be evaluated automatically: Is there a recent backup? Are we in a maintenance window? Is there enough disk space? Is the service healthy? Policy-as-code means the rules are version-controlled, auditable, and consistently enforced — no more "I thought it was OK to patch during business hours."
====

Before writing the playbook, understand the policies that govern patching at Meridian Financial.

. From the terminal, review the patch policy definition:
+
[source,role="execute"]
----
cat /var/lib/awx/projects/defend-contain-comply/lab/policies/patch-policy.yml
----
+
Note the four automatable gates: maintenance window, backup verification, disk space, and service health. The fifth gate (change approval) is enforced by an AAP approval node.

. Verify the current backup state (simulated — create the marker):
+
[source,role="execute"]
----
sudo touch /var/log/last-backup-timestamp
----
+
This simulates a successful backup completing within the policy window.

. Verify OPA is running and the patch policy is loaded:
+
[source,role="execute"]
----
curl -s http://central:8181/v1/policies | python3 -c "
import sys, json
data = json.load(sys.stdin)
for p in data.get('result', []):
    print(f\"  {p['id']}\")
"
----

=== Verify

The patch policy defines clear, automatable prerequisites. The backup marker exists and OPA has the `dcc/patch_policy` rules loaded.

== Exercise 2: Write the Pre-Patch Policy Check Playbook

[NOTE.why]
====
*Why we do this:* Ansible gathers the facts, OPA makes the decision. The playbook collects real system state — backup age, disk space, service health — and sends it to OPA for evaluation against the organization's patch policy. If any gate fails, patching is blocked with a clear reason. This separation of concerns means security teams own the policy (in OPA), platform teams own the automation (in Ansible), and neither can override the other without a visible audit trail.
====

You will write a playbook that gathers system facts, sends them to OPA for evaluation, and blocks patching if any policy gates fail. A skeleton file with detailed instructions has been provided.

. Examine the exercise skeleton:
+
[source,role="execute"]
----
cat /var/lib/awx/projects/defend-contain-comply/lab/playbooks/contain/pre-patch-check-exercise.yml
----
+
Read the comments carefully — they describe the OPA input payload format and the expected response structure.

. Copy the exercise file to work on it:
+
[source,role="execute"]
----
cp /var/lib/awx/projects/defend-contain-comply/lab/playbooks/contain/pre-patch-check-exercise.yml \
   /var/lib/awx/projects/defend-contain-comply/lab/playbooks/contain/my-pre-patch-check.yml
----

. Edit the file and complete all the TODO tasks:
+
[source,role="execute"]
----
vi /var/lib/awx/projects/defend-contain-comply/lab/playbooks/contain/my-pre-patch-check.yml
----
+
[TIP]
====
The key steps are:

. Use `ansible.builtin.stat` to get the backup marker's mtime, then calculate age in hours
. Use `ansible.builtin.systemd` to check httpd's ActiveState
. Get free disk space from `ansible_mounts` (filter for `mount == "/"`, divide `size_available` by 1073741824)
. POST the four values to `http://central:8181/v1/data/dcc/patch_policy/result` using `ansible.builtin.uri`
. Use `ansible.builtin.fail` with `when: not _opa_response.json.result.allow_patch` to enforce the policy
====

. Test your playbook:
+
[source,role="execute"]
----
cd /var/lib/awx/projects/defend-contain-comply/lab
ansible-playbook playbooks/contain/my-pre-patch-check.yml -i inventory/workshop.yml
----

. If you get stuck, the solution is available:
+
[source,role="execute"]
----
cat /var/lib/awx/projects/defend-contain-comply/lab/playbooks/contain/pre-patch-check.yml
----

=== Verify

Your playbook should:

* Query OPA and display all gate results (PASS/FAIL)
* Complete successfully when all gates pass
* Show the number of passed gates (4/4)

== Exercise 3: Build the Patching Workflow in AAP Controller

[NOTE.why]
====
*Why we do this:* The workflow enforces a sequence that can't be shortcut: policy check → human approval → patch → verify → report. The approval node is the Change Advisory Board in code — it pauses the workflow until an authorized person explicitly approves. Automation handles the tedious parts (gathering evidence, applying patches, running verification) while humans stay in the loop for the decision that carries risk. If the policy check fails, the workflow never reaches the approval node.
====

Now construct the workflow that chains the policy check, approval, patching, and reporting.

. In AAP Controller, navigate to **Resources → Templates** and click **Add → Add workflow template**:
+
* Name: `Policy-Gated Patching`
* Organization: `Default`
* Inventory: `DCC Workshop`
* Click **Save**

. Open the **Visualizer** and add workflow nodes in this order:
+
[cols="1,2,2"]
|===
| Step | Node Type | Template

| 1 | Job Template | Pre-Patch Check
| 2 | Approval | Name: "Approve Patch Application", Timeout: 900s
| 3 | Job Template | Patch System
| 4 | Job Template | Post-Patch Verify
| 5 | Job Template | Report Compliance
|===

. Connect the nodes with **On Success** links:
+
* Pre-Patch Check → *on success* → Approve Patch Application
* Approve Patch Application → *on approve* → Patch System
* Patch System → *on success* → Post-Patch Verify
* Post-Patch Verify → *on success* → Report Compliance

. Click **Save** in the Visualizer.

=== Verify

The workflow visualizer should show five nodes connected in a linear chain with an approval gate between the policy check and the patch application.

== Exercise 4: Execute the Patching Workflow

[NOTE.why]
====
*Why we do this:* Running the workflow end-to-end proves the full chain works: policy evaluation, human approval, patch application, post-patch verification, and compliance reporting — all as a single auditable unit of work. The `set_stats` data flowing between nodes creates a machine-readable audit trail that compliance teams can query.
====

. Launch the workflow by clicking **Launch**.

. The workflow will pause at the Approval node. Click the **Approval** node and select **Approve**.
+
This simulates the Change Advisory Board sign-off.

. Watch the workflow proceed through the remaining nodes:
* Patch System — installs `httpd-2.4.62-1.el9`
* Post-Patch Verify — confirms CVE is resolved
* Report Compliance — produces HTML artifact

. Once complete, verify from the terminal:
+
[source,role="execute"]
----
rpm -q httpd
----
+
Expected: `httpd-2.4.62-1.el9` (or newer).

. Confirm the CVE is no longer pending:
+
[source,role="execute"]
----
yum updateinfo list cves 2>/dev/null | grep CVE-2024-38476 || echo "CVE-2024-38476: RESOLVED"
----

=== Verify

* `httpd` is upgraded to the patched version
* CVE-2024-38476 no longer appears in pending advisories
* Workflow completed all nodes successfully

== Exercise 5: Review the Compliance Report

[NOTE.why]
====
*Why we do this:* Auditors don't accept "we ran the playbook" as evidence. They need timestamped proof of what was done, what changed, and what the system state is now. Generating the compliance report as a workflow step means evidence collection is automatic and contemporaneous — it captures the exact state at the moment of remediation, not a reconstruction days later. This is the difference between "we think we patched it" and "here's the proof."
====

The final workflow node generated an HTML compliance report.

. From the terminal, check the report was generated:
+
[source,role="execute"]
----
ls -la /tmp/compliance-reports/
----

. View the report content:
+
[source,role="execute"]
----
cat /tmp/compliance-reports/compliance-report-$(date +%Y-%m-%d).html | grep -A2 "Advisory Applied"
----

. In AAP Controller, check the workflow job → **Extra Vars / Artifacts** panel:
+
The `set_stats` data from each job shows the complete audit trail:
* `pre_patch_policy_passed: true`
* `patch_applied: true`
* `cve_resolved: true`
* `selinux_enforcing: true`

=== Verify

The compliance report contains:

* CVE identifier and advisory number
* Package version before and after
* SELinux status
* Firewall configuration
* Complete remediation timestamp

== Exercise 6: Test Policy Violation (Optional)

[NOTE.why]
====
*Why we do this:* A policy gate is only valuable if it actually blocks. Removing the backup marker simulates a real scenario — someone tries to patch a system that hasn't been backed up recently. The pre-patch check fails with a specific, actionable message. This demonstrates that the automation is safe by default: it won't take a risky action just because someone clicked "run."
====

Demonstrate what happens when policy gates fail.

. Remove the backup marker to simulate a stale backup:
+
[source,role="execute"]
----
sudo rm -f /var/log/last-backup-timestamp
----

. In AAP Controller, launch just the "Pre-Patch Check" job template.

. Observe the failure: OPA returns `allow_patch: false` and your playbook reports which gate was violated.

. Restore the marker for subsequent exercises:
+
[source,role="execute"]
----
sudo touch /var/log/last-backup-timestamp
----

=== Verify

The pre-patch check fails with a clear message identifying the violated policy gate.
